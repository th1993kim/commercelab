# Commerce Lab

> 커머스 주문 시스템에서 발생하는 주문, 재고, 결제, 비동기 이벤트, 장애 복구, 부하 테스트 문제를 직접 실험하고 문서화하는 개인 백엔드 프로젝트입니다.

## 1. 프로젝트 소개

Commerce Lab은 커머스 도메인의 핵심 흐름인 **주문 → 재고 예약 → 결제 승인 → 주문 확정 → 배송/정산 이벤트 처리**를 직접 구현하며, 백엔드 시스템 설계와 운영 관점의 문제를 학습하기 위한 개인 장기 프로젝트입니다.

이 프로젝트는 단순한 쇼핑몰 CRUD 구현이 아니라, 실제 서비스 환경에서 자주 발생할 수 있는 아래 문제들을 작은 규모에서 재현하고 개선하는 것을 목표로 합니다.

- 동시 주문 상황에서의 재고 정합성 문제
- 결제 승인, 실패, 취소, timeout 처리
- 중복 주문 및 중복 결제 방지
- Kafka 기반 비동기 이벤트 처리
- Redis를 활용한 재고 선점 및 캐싱 전략
- 장애 상황에서의 보상 트랜잭션과 재처리
- 부하 테스트를 통한 병목 분석
- Modular Monolith에서 MSA로 확장 가능한 구조 설계

본 프로젝트는 이직 준비 기간 동안 백엔드 시스템 설계와 운영 관점의 문제를 체계적으로 학습하기 위한 개인 실험 프로젝트이며, 실제 운영 서비스 경험과는 구분됩니다.

## 2. 프로젝트 목표

이 프로젝트의 목표는 다음과 같습니다.

### 2.1 도메인 목표

- 커머스 주문 흐름을 상태 기반으로 모델링합니다.
- 주문, 재고, 결제 도메인의 책임을 분리합니다.
- 결제 실패, 재고 부족, 주문 취소 등 예외 상황을 명확히 처리합니다.
- Fake PG 시스템을 통해 실제 PG와 유사한 결제 흐름을 시뮬레이션합니다.

### 2.2 기술 목표

- Java/Kotlin, Spring Boot 기반의 백엔드 시스템을 설계합니다.
- JPA와 MySQL을 활용해 트랜잭션과 데이터 정합성을 다룹니다.
- Redis를 활용해 동시성 제어와 재고 선점 전략을 실험합니다.
- Kafka를 활용해 주문 이후의 배송, 알림, 정산 처리를 비동기로 분리합니다.
- k6를 활용해 부하 테스트를 수행하고 병목을 분석합니다.
- Prometheus, Grafana 기반의 모니터링 환경을 구성합니다.
- 장애 시나리오를 정의하고 복구 전략을 문서화합니다.

### 2.3 학습 목표

- MSA를 무작정 적용하기보다, 먼저 Modular Monolith로 도메인 경계를 안정화합니다.
- 이후 결제, 재고 등 분리 가치가 높은 모듈을 단계적으로 서비스로 분리합니다.
- 기술 선택의 이유와 트레이드오프를 ADR 문서로 남깁니다.
- 성능 개선 전후 결과를 수치 기반으로 기록합니다.

## 3. 핵심 시나리오

### 3.1 주문 생성

1. 사용자가 상품을 주문합니다.
2. 주문 요청에는 상품 ID, 수량, 주문 요청 식별자가 포함됩니다.
3. 주문 서비스는 주문을 생성하기 전에 재고 예약을 요청합니다.
4. 재고 예약에 성공하면 결제를 요청합니다.
5. 결제 승인에 성공하면 주문을 확정합니다.
6. 주문 확정 이후 배송, 알림, 정산 이벤트를 발행합니다.

### 3.2 결제 승인

1. 주문 서비스가 결제 서비스에 결제 승인을 요청합니다.
2. 결제 서비스는 결제 요청 이력을 생성합니다.
3. 결제 서비스는 Fake PG 서비스에 승인 요청을 보냅니다.
4. Fake PG는 성공, 실패, timeout 등의 응답을 반환합니다.
5. 결제 결과에 따라 주문 상태를 변경합니다.

### 3.3 주문 취소

1. 사용자가 주문 취소를 요청합니다.
2. 결제 승인 상태라면 결제 취소를 요청합니다.
3. 결제 취소에 성공하면 재고를 복구합니다.
4. 주문 상태를 취소로 변경합니다.
5. 취소 이벤트를 발행합니다.

### 3.4 장애 상황

프로젝트에서는 다음과 같은 장애 상황을 실험합니다.

- 결제 승인 요청 중 Fake PG timeout 발생
- 결제 성공 후 주문 상태 변경 실패
- 재고 예약 성공 후 결제 실패
- Kafka 메시지 중복 소비
- Kafka consumer 장애
- Redis 장애
- DB connection pool 고갈
- 동일 상품에 대한 대량 동시 주문 요청

## 5. 아키텍처 방향

### 5.1 초기 구조: Modular Monolith

초기에는 하나의 Spring Boot 애플리케이션으로 시작합니다.

단, 내부 모듈은 명확히 분리합니다.

```text
commerce-lab
 ├── product
 ├── inventory
 ├── order
 ├── payment
 ├── delivery
 └── settlement
```

각 모듈은 패키지만 나눈 구조가 아니라, 도메인 모델과 애플리케이션 서비스, 저장소 인터페이스, 외부 연동 어댑터를 분리해 독립적인 책임을 갖도록 설계합니다.

초기 단계에서는 모듈 간 호출을 같은 프로세스 안에서 수행하되, 다음 원칙을 지킵니다.

- 다른 도메인의 테이블을 직접 수정하지 않습니다.
- 다른 도메인의 내부 구현 클래스를 직접 참조하지 않습니다.
- 모듈 간 통신은 명시적인 command/query/event 인터페이스를 통해 수행합니다.
- 트랜잭션 경계는 유스케이스 단위로 명확히 정의합니다.
- 외부 시스템으로 분리될 가능성이 높은 결제, 재고, 정산 모듈은 인터페이스 기반으로 격리합니다.

### 5.2 확장 구조: MSA 전환 실험

MSA 전환은 프로젝트 초기 목표가 아니라, Modular Monolith에서 도메인 경계와 장애 처리 방식이 검증된 이후의 실험 단계로 둡니다.

서비스 분리 후보는 다음 기준으로 판단합니다.

| 분리 후보 | 분리 이유 | 주요 리스크 |
|---|---|---|
| inventory-service | 주문 트래픽 집중, 재고 동시성 제어 독립 필요 | 재고 예약/복구의 최종 정합성 |
| payment-service | PG 연동, timeout, 취소, 정산과의 책임 분리 | 중복 결제 방지, 주문 상태와의 불일치 |
| order-service | 주문 상태 머신과 사용자 요청 처리 중심 | 분산 트랜잭션 회피, 보상 처리 |
| settlement-service | 주문 확정 이후 비동기 정산 처리 | 이벤트 누락, 중복 정산 |

MSA 전환 시에는 분산 트랜잭션을 직접 사용하지 않고, Saga와 Outbox Pattern을 기반으로 최종 정합성을 맞추는 방향을 실험합니다.

```text
client
  |
  v
api-gateway / reverse-proxy
  |
  +--> order-service
  |      |
  |      +--> order-db
  |
  +--> inventory-service
  |      |
  |      +--> inventory-db
  |
  +--> payment-service
         |
         +--> payment-db

kafka
  +--> delivery-consumer
  +--> settlement-consumer
  +--> notification-consumer
```

MSA 전환 후 검증할 항목은 다음과 같습니다.

- 서비스 간 동기 호출 실패 시 timeout, retry, fallback 정책
- 주문 생성 중 재고 예약 성공 후 결제 실패 시 보상 처리
- 결제 승인 성공 후 주문 확정 실패 시 재처리 전략
- Kafka 메시지 중복 발행/중복 소비에 대한 idempotency 보장
- 각 서비스별 DB 분리 후 데이터 조회 모델 구성
- 장애 발생 시 사용자가 보게 되는 주문 상태 정의

## 6. 우선 구현 범위

이 프로젝트는 모든 커머스 기능을 넓게 구현하기보다, 백엔드 시스템 설계 역량을 보여줄 수 있는 핵심 흐름을 먼저 완성합니다.

### 6.1 1차 구현: 주문-재고-결제 핵심 플로우

- 상품 등록 및 조회
- 재고 등록, 차감, 복구
- 주문 생성 API
- 주문 취소 API
- Fake PG 승인/취소 API
- 주문 상태 조회 API
- 주문 요청 idempotency key 처리
- 결제 요청 idempotency key 처리

주문 상태는 다음과 같이 시작합니다.

```text
CREATED
RESERVED
PAYMENT_REQUESTED
PAID
CONFIRMED
CANCEL_REQUESTED
CANCELED
FAILED
```

### 6.2 2차 구현: 동시성 제어 비교

동일 상품에 대량 주문이 동시에 들어오는 상황을 재현하고, 재고 정합성과 처리량을 비교합니다.

비교 대상은 다음과 같습니다.

- 단순 DB update
- DB pessimistic lock
- DB optimistic lock
- Redis 기반 재고 선점

각 방식마다 다음 결과를 문서화합니다.

- 성공 주문 수
- 실패 주문 수
- 최종 재고 수량
- 평균 응답 시간
- p95 응답 시간
- deadlock 또는 retry 발생 여부
- 구현 복잡도와 운영 리스크

### 6.3 3차 구현: 결제 장애 처리

Fake PG를 통해 실제 결제 연동에서 자주 발생하는 실패를 재현합니다.

- 결제 승인 성공
- 결제 승인 실패
- 결제 승인 timeout
- 결제 승인 결과 조회
- 결제 취소 성공
- 결제 취소 실패
- 중복 승인 요청
- 중복 취소 요청

핵심 검증 포인트는 "사용자에게 실패로 보였지만 실제 PG에서는 결제가 성공한 경우"를 어떻게 복구할지입니다.

### 6.4 4차 구현: 이벤트 기반 후속 처리

주문 확정 이후의 배송, 알림, 정산은 Kafka 이벤트로 분리합니다.

- `OrderConfirmedEvent` 발행
- `OrderCanceledEvent` 발행
- 배송 consumer
- 정산 consumer
- 알림 consumer
- consumer 중복 처리 방어
- 실패 이벤트 retry
- DLQ 또는 실패 테이블 기록

이 단계에서는 Outbox Pattern을 적용해 DB commit과 이벤트 발행 사이의 불일치를 줄입니다.

### 6.5 5차 구현: 부하 테스트와 관측

k6로 시나리오 기반 부하 테스트를 수행하고, Prometheus/Grafana로 병목을 관찰합니다.

- 단일 상품 동시 주문
- 여러 상품 분산 주문
- 결제 timeout 비율이 포함된 주문
- Kafka consumer 지연 상황
- DB connection pool 고갈 상황

결과 문서에는 개선 전후 수치를 함께 남깁니다.

## 9. AI 활용 원칙

이 프로젝트에서는 AI를 개발 보조 도구로 활용하되, 최종 판단과 검증 책임은 개발자가 갖습니다.

### 9.1 활용 범위

- 요구사항 정리와 README, ADR, 실험 문서 초안 작성에 활용합니다.
- 코드 구조 설계, 구현 아이디어, 예외 케이스 탐색에 활용합니다.
- 테스트 케이스, 장애 시나리오, 부하 테스트 관점 도출에 활용합니다.
- 코드 리뷰 보조 도구로 활용해 버그 가능성, 유지보수성, 누락된 테스트를 점검합니다.

### 9.2 검증 원칙

- AI가 생성한 코드는 그대로 신뢰하지 않고 직접 읽고 이해한 뒤 반영합니다.
- 중요한 변경은 테스트, 로그, 실행 결과, 수치 등으로 검증합니다.
- 프로젝트의 기존 코드 스타일, 패키지 구조, 도메인 경계를 우선합니다.
- 설계 결정이나 기술 선택의 이유는 README, ADR, 실험 문서에 기록합니다.

### 9.3 보안 원칙

- API key, 비밀번호, 토큰, 인증 정보는 AI에 입력하지 않습니다.
- 실제 사용자 데이터, 결제 정보, 개인정보를 AI에 입력하지 않습니다.
- 외부에 공개하면 안 되는 로그나 설정 파일은 필요한 부분만 익명화해 사용합니다.

### 9.4 학습 원칙
