# Commerce Lab

> 커머스 주문 시스템에서 발생하는 주문, 재고, 결제, 비동기 이벤트, 장애 복구, 부하 테스트 문제를 직접 실험하고 문서화하는 개인 백엔드 프로젝트입니다.

## 1. 프로젝트 소개

Commerce Lab은 커머스 도메인의 핵심 흐름인 **주문 → 재고 예약 → 결제 승인 → 주문 확정 → 배송/정산 이벤트 처리**를 직접 구현하며, 백엔드 시스템 설계와 운영 관점의 문제를 학습하기 위한 개인 장기 프로젝트입니다.

이 프로젝트는 단순한 쇼핑몰 CRUD 구현이 아니라, 실제 서비스 환경에서 자주 발생할 수 있는 아래 문제들을 작은 규모에서 재현하고 개선하는 것을 목표로 합니다.

- 동시 주문 상황에서의 재고 정합성 문제
- 결제 승인, 실패, 취소, timeout 처리
- 중복 주문 및 중복 결제 방지
- Kafka 기반 비동기 이벤트 처리
- Redis를 활용한 재고 선점 및 캐싱 전략
- 장애 상황에서의 보상 트랜잭션과 재처리
- 부하 테스트를 통한 병목 분석
- Modular Monolith에서 MSA로 확장 가능한 구조 설계

본 프로젝트는 개인 학습 및 포트폴리오 목적의 프로젝트이며, 실제 운영 서비스 경험과는 구분됩니다.

## 2. 프로젝트 목표

이 프로젝트의 목표는 다음과 같습니다.

### 2.1 도메인 목표

- 커머스 주문 흐름을 상태 기반으로 모델링합니다.
- 주문, 재고, 결제 도메인의 책임을 분리합니다.
- 결제 실패, 재고 부족, 주문 취소 등 예외 상황을 명확히 처리합니다.
- Fake PG 시스템을 통해 실제 PG와 유사한 결제 흐름을 시뮬레이션합니다.

### 2.2 기술 목표

- Java/Kotlin, Spring Boot 기반의 백엔드 시스템을 설계합니다.
- JPA와 MySQL을 활용해 트랜잭션과 데이터 정합성을 다룹니다.
- Redis를 활용해 동시성 제어와 재고 선점 전략을 실험합니다.
- Kafka를 활용해 주문 이후의 배송, 알림, 정산 처리를 비동기로 분리합니다.
- k6를 활용해 부하 테스트를 수행하고 병목을 분석합니다.
- Prometheus, Grafana 기반의 모니터링 환경을 구성합니다.
- 장애 시나리오를 정의하고 복구 전략을 문서화합니다.

### 2.3 학습 목표

- MSA를 무작정 적용하기보다, 먼저 Modular Monolith로 도메인 경계를 안정화합니다.
- 이후 결제, 재고 등 분리 가치가 높은 모듈을 단계적으로 서비스로 분리합니다.
- 기술 선택의 이유와 트레이드오프를 ADR 문서로 남깁니다.
- 성능 개선 전후 결과를 수치 기반으로 기록합니다.

## 3. 핵심 시나리오

### 3.1 주문 생성

1. 사용자가 상품을 주문합니다.
2. 주문 요청에는 상품 ID, 수량, 주문 요청 식별자가 포함됩니다.
3. 주문 서비스는 주문을 생성하기 전에 재고 예약을 요청합니다.
4. 재고 예약에 성공하면 결제를 요청합니다.
5. 결제 승인에 성공하면 주문을 확정합니다.
6. 주문 확정 이후 배송, 알림, 정산 이벤트를 발행합니다.

### 3.2 결제 승인

1. 주문 서비스가 결제 서비스에 결제 승인을 요청합니다.
2. 결제 서비스는 결제 요청 이력을 생성합니다.
3. 결제 서비스는 Fake PG 서비스에 승인 요청을 보냅니다.
4. Fake PG는 성공, 실패, timeout 등의 응답을 반환합니다.
5. 결제 결과에 따라 주문 상태를 변경합니다.

### 3.3 주문 취소

1. 사용자가 주문 취소를 요청합니다.
2. 결제 승인 상태라면 결제 취소를 요청합니다.
3. 결제 취소에 성공하면 재고를 복구합니다.
4. 주문 상태를 취소로 변경합니다.
5. 취소 이벤트를 발행합니다.

### 3.4 장애 상황

프로젝트에서는 다음과 같은 장애 상황을 실험합니다.

- 결제 승인 요청 중 Fake PG timeout 발생
- 결제 성공 후 주문 상태 변경 실패
- 재고 예약 성공 후 결제 실패
- Kafka 메시지 중복 소비
- Kafka consumer 장애
- Redis 장애
- DB connection pool 고갈
- 동일 상품에 대한 대량 동시 주문 요청
